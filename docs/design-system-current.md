# Rehab Tracker ŌĆö Current Design System Reference

Snapshot of the **actually implemented** UI/UX, generated by reading the codebase directly (not the other way around). Use this to recreate or extend mockups that match production, not an idealized design.

- **Stack**: React 19 + TypeScript, React Router 7, Tailwind CSS v4 (CSS-first config via `@theme`, no `tailwind.config.js`), Dexie (IndexedDB) for local-only storage, Recharts for the pain-trend chart, FontAwesome (`free-solid-svg-icons`) for icons, `clsx` for conditional classes.
- **Theme file**: `src/index.css` ŌĆö all design tokens live here as CSS custom properties inside `@theme { ... }`. There is no separate JS theme/constants file.
- **Dark mode only**: `:root { color-scheme: dark; }` is hardcoded. There's an unused `useIsDarkMode` hook (`src/lib/useIsDarkMode.ts`) that watches `prefers-color-scheme` but nothing in the app consumes it ŌĆö no light theme exists today. Treat this app as single-theme (dark) when reproducing mockups.
- **App name shown in UI**: "Rehab Tracker" (in both the sidebar and mobile header, and the `<title>`), even though the repo/package is named `injury-tracker`.

## 1. Design Tokens

All defined in `src/index.css` under `@theme`. Values are **oklch**, given here with the raw token plus a note on where it's used. (oklch, not hex ŌĆö reproduce with oklch() in CSS; Claude Design tooling that needs hex can convert but should not "round" these to nearest named color.)

### 1.1 Color ŌĆö Surfaces (dark, warm-neutral near-black scale)

| Token | Value | Usage |
|---|---|---|
| `--color-canvas-page` | `oklch(0.07 0.008 60)` | Outermost page background (`body`) |
| `--color-canvas` | `oklch(0.10 0.009 60)` | Main content background inside the app shell frame |
| `--color-canvas-sidebar` | `oklch(0.14 0.009 60)` | Desktop sidebar background |
| `--color-surface` | `oklch(0.21 0.012 60)` | Default card background (`Card`) |
| `--color-surface-raised` | `oklch(0.24 0.013 60)` | Modal background, hover states, "raised" panels (e.g. select-mode toolbar) |
| `--color-control` | `oklch(0.28 0.015 60)` | Secondary button bg, slate badge bg, IconButton neutral bg |
| `--color-input` | `oklch(0.15 0.01 60)` | Input/Select/Textarea backgrounds (darker than surface ŌĆö recessed look) |

All share hue `60` (warm/brown-neutral) and low chroma (0.008ŌĆō0.015) ŌĆö a monochrome warm-gray scale distinguished only by lightness. This is the backbone of the "elevation" system: page < canvas < sidebar < surface < surface-raised < control, each step roughly +0.03ŌĆō0.07 lightness.

### 1.8 Spacing

No custom spacing scale is declared ŌĆö the app uses **Tailwind's default 4px-based scale** (`space-y-2` = 8px, etc.) supplemented heavily by **arbitrary bracket values** that still land near-4px increments (e.g. `px-[14px]`, `py-[9px]`, `p-[18px]`, `p-[22px]`). There's no dedicated `--spacing-*` token set in `@theme`.

Common recurring spacing values (component-level, not systematized):
- Page-level vertical rhythm: `space-y-5` (20px) between major page sections
- Card internal padding: `p-[18px]` (Card default), `p-[22px]` (Modal)
- Compact list-item padding: `px-3 py-[9px]` (Input, Select, Textarea, remedy/trigger list rows), `px-[14px] py-3` (timeline items)
- Small gaps between inline elements: `gap-1.5` (6px) icon/badge clusters, `gap-2` (8px) button groups, `gap-2.5` (10px) action rows

### 1.9 Border Radius

No `--radius-*` tokens ŌĆö every component hand-picks a pixel value in Tailwind arbitrary-value syntax. The **de facto scale** in use today:

| Value | Where |
|---|---|
| `rounded-md` (6px) | `IconButton` sm size |
| `rounded-[9px]` | `Button` sm size |
| `rounded-[10px]` | `Button` md size, `Input`/`Select`/`Textarea`, `IconButton` md size, nav links, TriggerList/RemedyList row items |
| `rounded-xl` (12px) | `Button` lg size, dashboard FAB-adjacent... (also `rounded-lg`/`rounded-xl` used loosely e.g. `EntityForm` wrapper `rounded-lg` = 8px) |
| `rounded-[12px]` | InjuryCard select-mode toolbar |
| `rounded-[14px]` | LogEntryModal per-injury group wrapper |
| `rounded-[16px]` | `Card` |
| `rounded-[18px]` | `Modal` panel |
| `rounded-[20px]` | Desktop AppShell outer frame |
| `rounded-full` | `Badge`, `TogglePill`, range-slider thumb/track, FAB, chart range-toggle buttons |

This is a genuine inconsistency (see ┬¦6) ŌĆö treat 9/10/12/14/16/18/20px + `full` as the working palette, but don't assume a formal 2px or 4px step system; it's closer to "increases roughly with component size/prominence."

| Where | Value |

Both custom shadows are large, soft, downward-offset "floating panel" shadows ŌĆö negative spread pulls them tight to the element edge horizontally while still reading as a big soft drop below.

## 3. Layout Patterns

### 3.1 App shell (`AppShell.tsx`)

- Outer wrapper: full-height, `bg-canvas-page`, centered content up to `max-w-[1400px]`.
- **Desktop (`lg:` Ōēź1024px)**: two-column CSS grid `lg:grid-cols-[220px_1fr]` inside a single bordered/rounded/shadowed frame (`rounded-[20px] border border-subtle bg-canvas shadow-[...]`, `overflow-hidden`). Left column is a fixed 220px sidebar (`bg-canvas-sidebar`) with the wordmark + vertical nav (Dashboard, Settings). Right column holds the sticky-less main content.
- **Mobile/tablet (below `lg`)**: sidebar is hidden; instead a sticky top header bar (`sticky top-0 z-10`, blurred translucent bg `bg-canvas/90 backdrop-blur`) shows the wordmark + horizontal nav pills. A floating circular "+" FAB (`fixed bottom-6 right-6`, 56├Ś56px, `bg-accent`) opens the log-entry modal; it's hidden on desktop (`lg:hidden`) since desktop presumably relies on in-page "Log Entry" buttons/shortcuts instead.
- Main content padding: `px-4 py-5` on mobile, `sm:px-6`, `lg:p-6` on desktop, content capped at `max-w-6xl` on mobile/tablet but unconstrained (`lg:max-w-none`) on desktop since the outer shell already caps width.
- Active nav link styling (both mobile and desktop): `bg-accent-soft font-semibold text-accent-soft-text`; inactive: muted text, no background.
- `BackupBanner` is wired into `main` but **commented out** ŌĆö see ┬¦6.

### 3.2 Page-level conventions

Every page root is `<div className="space-y-5">` (20px vertical rhythm between sections). Every page's `<h1>` is `font-heading text-2xl font-semibold text-ink`. There is no page-level "PageHeader" component ŌĆö the title/actions row is hand-rolled per page (`DashboardPage`, `InjuryDetailPage`) or just a bare `<h1>` (`SettingsPage`, `InjuryFormPage`).

### 3.3 Grids / breakpoints

Only two Tailwind breakpoints are actively used: `sm:` (640px, mostly for the Modal's dialog-vs-sheet switch and page padding) and `lg:` (1024px, for the entire shell layout switch and content grids). No `md:` or `xl:` usage found ŌĆö this is effectively a 2-breakpoint (mobile / desktop) responsive system, not a full 4-5 tier grid system.

- Dashboard injury grid: `grid-cols-1 lg:grid-cols-2` (single column until desktop, then 2-up)
- Injury detail page: `grid lg:grid-cols-[2fr_1fr]` ŌĆö main column (chart + timeline) vs sidebar column (remedies + triggers), stacked on mobile

## 4. Key Screens

### 4.1 Dashboard (`/`, `DashboardPage.tsx`)
- Header row: "Your injuries" `h1` + action buttons. Normal mode: `Select` (secondary) + `Add Injury` (primary, with `N` kbd hint). Select mode: swaps to a single `Cancel` (ghost, `Esc` hint) button.
- Status filter row: "Filter by:" label + `TogglePill` row (Active/Monitoring/Resolved), multi-selectable, all selected by default.
- When in select mode and Ōēź0 selected: an extra toolbar `Card`-like strip (`rounded-[12px] bg-surface-raised`) shows "`N` selected" + a `danger` "Delete selected" button (disabled at 0).
- Body states (in priority order): loading (`LoadingŌĆ”` muted text) ŌåÆ zero injuries at all (`No injuries tracked yet...`) ŌåÆ zero matching filter (`No injuries match the selected filters.`) ŌåÆ grid of `InjuryCard`s.
- `n` keyboard shortcut jumps to new-injury form (disabled while in select mode); `Escape` exits select mode.

### 4.2 Injury Detail (`/injuries/:id`, `InjuryDetailPage.tsx`)
- Loading state: plain `LoadingŌĆ”` text. Not-found state: plain `Injury not found.` text (no dedicated illustration/empty-state component ŌĆö same minimal pattern used everywhere).
- Header: injury name (`h1`) + status badge top-right; optional description paragraph (`max-w-3/5`, muted) below.
- Action row: `Log Entry` (primary, `L` hint) + edit `IconButton` (pencil) on the left; delete `IconButton` (trash, danger tone) on the right ŌĆö deliberately separated to opposite ends of the row to reduce accidental-delete risk.
- Two-column body (desktop): left = `PainTrendChart` then `LogTimeline`; right = `RemedyList` then `TriggerList`. Right column uses `lg:self-start` so it doesn't stretch to match the (usually taller) left column.
- `l` keyboard shortcut opens the log modal pre-scoped to this injury.

### 4.3 Injury Form (`/injuries/new`, `/injuries/:id/edit`, `InjuryFormPage.tsx`)
- Full-page (not modal) form, reused for both create/edit via `isEdit` flag. Status field only shown when editing (new injuries default to `active` implicitly). Submit label switches "Submit" (create) vs "Save" (edit).

### 4.4 Log Entry Modal (global, `LogEntryModal.tsx`)
- Opened via FAB, `InjuryCard`'s "Log Entry" button, detail-page "Log Entry" button, or `l`/`n`-adjacent flows.
- Structure: `InjurySelector` (which injuries does this entry cover) ŌåÆ for each selected injury, a bordered sub-card (`rounded-[14px] border border-subtle p-[14px]`) containing that injury's name + its own `PainSlider`, `PainFrequencySlider`, `RemedyCheckboxGroup`, `TriggerCheckboxGroup` ŌåÆ shared `When` datetime field ŌåÆ shared `Notes` textarea.
- This is a "session" concept: one submit creates one `LogEntry` per selected injury sharing the same timestamp/notes but independent pain data ŌĆö reflected in `createLogSession`.
- Footer: `Submit` (primary, disabled until Ōēź1 injury selected) + `Cancel` (ghost). Note the footer button *order* here is Submit-then-Cancel, whereas `LogEntryEditModal` and most other dual-button forms use Cancel-then-Save order ŌĆö a minor inconsistency (see ┬¦6).

### 4.5 Log Entry Edit Modal (`LogEntryEditModal.tsx`)
- Same field set as the creation modal but for a single existing entry, opened by clicking the pencil `IconButton` on a `LogTimelineItem`.

### 4.6 Settings (`/settings`, `SettingsPage.tsx`)
- Three stacked `Card`s: **Backup** (export/import, last-exported-at relative timestamp), **Seed data** (load/clear example data ŌĆö a dev/demo convenience), **Danger zone** (delete all data, red-tinted border override, red heading text).
- Transient success messages render as a green (`text-pain-green`) line above the cards, cleared on next action.
- Import uses a hidden native `<input type=file accept="application/json">` triggered by a visible `Import data` button (common a11y-friendly file-input pattern).

### 4.7 Not Found (`*`, `NotFoundPage.tsx`)
- Minimal centered text + "Back to dashboard" link. No illustration, no `AppShell`-independent layout ŌĆö still renders inside the shell.

### Interaction/state pattern summary (cross-cutting)
- **Loading**: always a plain `<p className="text-ink-muted">LoadingŌĆ”</p>` (or `LoadingŌĆ”` inline) ŌĆö no skeletons/spinners anywhere in the app.
- **Empty states**: always plain muted `<p>` text, phrased contextually (e.g. "No injuries tracked yet. Add one to start logging.", "No log entries yet.", "No rated entries in this range yet.") ŌĆö no icons or illustrations.
- **Destructive confirmation**: always the native browser `confirm()` ŌĆö no custom confirmation dialog component exists, for single delete, bulk delete, and settings' delete-all.
- **Transitions**: only `transition-colors` on interactive elements (buttons, links, pills, icon buttons) ŌĆö no layout, opacity, or transform transitions/animations anywhere (chart lines even set `isAnimationActive={false}` explicitly).
- **Keyboard shortcuts**: a real, consistent affordance across the app ŌĆö `Kbd` hints are rendered directly inside button labels wherever a shortcut exists (`N` new injury, `L` log entry, `Esc` cancel, `ŌīśS`/`Ctrl+S` save-in-forms). Implemented via `useKeyboardShortcut` (global single-key) and `useFormShortcuts` (form-scoped save/cancel).

## 6. Inconsistencies to Reconcile (don't copy forward)

1. **Two independent "danger red" definitions.** `Button`'s `danger` variant and `IconButton`'s `danger` tone (md size) both hardcode the identical trio `oklch(0.33 0.14 25)` / `oklch(0.37 0.14 25)` / `oklch(0.96 0.03 25)` inline rather than referencing a shared token ŌĆö and this red is visually distinct from `--color-pain-red-bg` (`oklch(0.30 0.1 25)`), which is *also* a "red" used for the same semantic concept (danger/severe) elsewhere. There are effectively two reds in the app that should probably be one token.
2. **`BackupBanner` is built but disabled.** `AppShell.tsx` line 63 has `{/* <BackupBanner /> */}` ŌĆö the component, its hook wiring, and its 14-day-reminder logic all exist and work, but it's commented out, so no banner currently ever renders. If reproducing "current" behavior faithfully, omit it; if asked to show the "intended" banner design, know it renders as a full-width amber notice below the shell header using entirely off-token amber colors (┬¦1.6) rather than the `--color-pain-amber*` tokens already in the theme.
3. **No border-radius scale ŌĆö every component invented its own value.** See ┬¦1.9: 6/9/10/12/14/16/18/20px + full, with no visible rule for which component gets which (e.g. `Button` lg is 12px but `Card`, which is visually larger, is 16px; `EntityForm`'s wrapper is a plain Tailwind `rounded-lg`=8px, smaller than any Button). Treat this as organic/unsystematic rather than a deliberate scale.
4. **Pain-tone color logic is duplicated, not shared.** `LogTimelineItem.tsx` has local `painTone()`/`freqTone()` functions mapping numeric thresholds to badge tones. `PainTrendChart.tsx` separately hardcodes its own `colors` object with visually-similar-but-independently-declared oklch values for the same intensity/frequency concepts (e.g. chart's `line: oklch(0.58 0.13 250)` matches `--color-accent` exactly by coincidence of copy-paste, but isn't referencing the token). A future pass should centralize "what color is pain level X" into one helper/token set.
5. **Modal footer button order is inconsistent between the two log modals.** `LogEntryModal`'s footer renders primary `Submit` **first**, then ghost `Cancel` second. `LogEntryEditModal`'s footer renders ghost `Cancel` **first**, then primary `Save` second ŌĆö the reverse order, despite the two modals sharing nearly identical field sets and being opened from adjacent flows (create vs. edit the same kind of entry). `InjuryForm` and `EntityForm` both use primary-then-ghost (matching `LogEntryModal`), so `LogEntryEditModal` is the outlier. Pick one order and apply it everywhere.
6. **`RemedyList` groups Relief vs. Long-term; `TriggerList` has no grouping.** Structurally near-duplicate components (list + inline edit `EntityForm` + archive `IconButton` + dashed "+ Add" button) but `RemedyList` wraps two `RemedyGroup` instances while `TriggerList` inlines the equivalent markup once, unfactored. A shared `EntityList`/`EntityGroup` component would remove this duplication.
7. **Dead/unused dark-mode hook.** `useIsDarkMode.ts` exists and correctly watches `prefers-color-scheme`, but has zero call sites. The app is hardcoded dark-only (`color-scheme: dark` in `:root`). Don't assume light-mode support exists; either wire the hook up or remove it.
8. **Font-loading is CDN-based, not self-hosted.** Both `Lora` and `Manrope` are pulled from `fonts.googleapis.com` at runtime rather than bundled as static assets ŌĆö fine for prototyping, but a mockup/offline tool should be aware the app has an external font dependency, not local font files.
